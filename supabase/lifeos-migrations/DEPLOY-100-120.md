## 2. Google: Gmail opnieuw koppelen

**Nodig voor:** AI-conceptantwoorden, archiveren, labelen.

De scope gaat van `gmail.readonly` naar **`gmail.modify`**. Je bestaande
koppeling heeft die niet, dus je moet één keer opnieuw koppelen. Tot dat moment
krijg je een nette "koppel opnieuw voor schrijfrecht"-melding in plaats van een
Google-403 — de pre-flight scope-check leest nu eindelijk het opgeslagen `bereik`
terug.

**Lees dit voordat je consent geeft.** `gmail.modify` mág ook versturen. Google
heeft geen scope die concepten toestaat maar versturen verbiedt. Tot nu toe was
"LifeOS verstuurt nooit namens jou" gegarandeerd door het ontbreken van de scope
— een slot. Vanaf nu is het een code-discipline: er ís geen send-pad in de code,
maar de scope zou het toelaten. Dat is de prijs van conceptantwoorden. Wil je die
prijs niet betalen, zeg het, dan draaien we de Gmail-schrijfacties terug en blijft
de inbox read-only.

De agenda-scope verandert niet.

## 3. Render — environment

| Var | Waarom | Status |
|---|---|---|
| `LIFEOS_TELEGRAM_ALLOWED_CHAT_ID` | **Nu verplicht.** Was fail-open (leeg = iedereen erlangs); nu fail-closed. | **Zetten, anders doet de bot niets** |
| `LIFEOS_TELEGRAM_CHAT_ID` | Waar de dagbriefing heen gaat | Nieuw |
| `CRON_SECRET` | Beveiligt `/api/cron/lifeos-briefing`, fail-closed | Zetten |
| `OAUTH_STATE_SECRET` | Agenda/mail koppelen (stond al open uit een eerdere ronde) | Nog steeds open |
| `TZ` | **`Europe/Amsterdam`** — harde eis | Controleren |

Over `TZ`: `vrije-blokken.ts` gebruikt lokale `setHours` voor het werkvenster.
Staat de server op UTC, dan plant het focusblok 's zomers je training om 06:00.
Dit is geen nieuwe eis, maar hij gaat nu wél iets doen.

Over de allowlist: die stond fail-open terwijl diezelfde bot autonoom afspraken in
je échte agenda schrijft — het webhook-secret bewijst "dit komt van Telegram",
niet "dit komt van Kane". Ken je je chat-id niet, zet dan tijdelijk
`LIFEOS_TELEGRAM_ALLOWED_CHAT_ID=leer-modus`: de bot meldt je chat-id terug en
voert verder **niets** uit (geen Whisper, geen Claude, geen schrijfactie). Zet
daarna het echte id.

**Render auto-deployt niet op een push.** Dashboard → service → Manual Deploy →
"Deploy latest commit".

## 4. GitHub — de dagbriefing aanzetten

1. Repo → Settings → Secrets → Actions → `CRON_SECRET` (zelfde waarde als in Render).
2. Actions → **LifeOS-dagbriefing** → Run workflow (één keer handmatig).
3. **Kijk of er écht een Telegram-bericht binnenkomt.**

Stap 3 is niet optioneel. Vita's kaart zegt pas dat hij je aantikt zodra er een
rij in `vita_briefingen` staat — het logboek van wat er daadwerkelijk bezorgd is.
Draait de cron niet, dan zegt de kaart eerlijk "Uit mezelf tik ik je nu nog niet
aan". Dat is met opzet: een cron die in de codebase staat is niet een cron die
draait.

De GitHub-cron draait op UTC: 07:00 in de zomer, 06:00 in de winter. Runs kunnen
uitlopen of overslaan — GitHub garandeert geen exacte tijd.

## Wat er daarna nog open staat

- **Twee keer dezelfde inplan-logica.** `taken/dagplan.ts` (adviseert) en
  `agenda/inplannen.ts` (schrijft) bevatten allebei een greedy first-fit over
  dezelfde vrije blokken. Twee agents bouwden het onafhankelijk. Geen bug — de
  contracten sluiten aan — maar dezelfde regel op twee plekken loopt uit elkaar.
- **`Foutmelding` zonder `opnieuw`** in `AgendaKaart`, `NieuweAfspraak`,
  `InboxKaart`, `BrainDumpKaart`, terwijl het component zelf zegt: *"Weglaten =
  geen weg terug. Doe dat niet."*
- **Rate limiting is per proces** (in-memory). Bij meerdere Render-instances geldt
  het per instance. Een rem tegen dubbelklikken, geen quotum.
- **De dagbriefing is deterministisch, geen LLM.** Blunter, maar een bericht dat
  ongevraagd je telefoon laat trillen moet je woord voor woord kunnen navertellen.
- **Business-domein is leeg** en blijft dat tot er een bron is. Er is nul
  business-data in dit systeem; de kaart zegt wat ervoor nodig zou zijn.
